# RAG-пайплайн ЧитАИ

## Назначение

Документ описывает архитектуру Retrieval-Augmented Generation (RAG) пайплайна, используемого в ЧитАИ для генерации ответов с прозрачным цитированием источников.

## 1. Общая схема

```
[Запрос] → [Препроцессинг + PII-фильтр] → [Эмбеддинг]
        → [Векторный поиск top-K]
        → [Reranking]
        → [Сборка контекста]
        → [Промпт + LLM]
        → [Постпроцессинг + проверка ссылок]
        → [Ответ + источники]
```

## 2. Этапы

### 2.1. Препроцессинг

- Нормализация (нижний регистр, удаление лишних пробелов).
- Проверка на PII (телефон, email, СНИЛС, паспорт, адрес) — в случае обнаружения PII в запросе, при необходимости — маскирование перед отправкой в LLM.
- Проверка по списку запрещённых тем (см. `05_Юр_пакет/08_этический_протокол.md`).

### 2.2. Эмбеддинг запроса

- На MVP: TF-IDF / BM25 (локально, на основе sklearn / rank_bm25).
- В продакшене: BGE-M3 (multilingual) или Yandex Embeddings.
- Размерность вектора: 1024 (BGE-M3) или 768 (Yandex Embeddings).

### 2.3. Векторный поиск

- На MVP: in-memory FAISS / линейный поиск по JSON.
- В продакшене: PostgreSQL + pgvector с HNSW-индексом.
- Параметр top-K: 12 (предварительный), затем после reranking — top-5.
- Метрика: cosine similarity.

### 2.4. Reranking (опционально)

- На MVP: сортировка по similarity score.
- В продакшене: cross-encoder (например, BGE-Reranker) для более точного скоринга.

### 2.5. Сборка контекста

- Из top-5 источников формируется контекст:
  - название произведения, автор, год;
  - текст фрагмента (длина 200–500 слов);
  - метаданные источника.
- Общий лимит контекста: 4096 токенов.
- Включается ссылка на оригинал каждого фрагмента.

### 2.6. Промпт

Системный промпт:

```
Вы — российский ИИ-куратор маршрутов чтения по фондам РГБ и НЭБ.
Отвечайте по-русски, академично, без англицизмов и эмодзи.
Используйте только информацию из предоставленного контекста.
В каждом ответе указывайте источники (название, автор, год).
Если в контексте нет информации для ответа — ответьте: «не нашёл данных в фондах».
Возрастной режим: {age_mode}.
```

Промпт пользователя:

```
Контекст:
{retrieved_chunks}

Запрос пользователя:
{user_query}

Ответ:
```

### 2.7. Вызов LLM

- На MVP: моковая генерация (детерминированные шаблоны на Python по ключевым словам и интентам).
- В продакшене:
  - YandexGPT 5 Pro (основной).
  - GigaChat MAX (резервный + для cross-проверки).
- Параметры: temperature 0.2, max_tokens 1024, top_p 0.9.

### 2.8. Постпроцессинг

- Проверка наличия ссылок на источники.
- Проверка отсутствия PII в ответе.
- Проверка на «галлюцинации»: если LLM дал ответ без подтверждения в контексте — ответ заменяется на «не нашёл данных в фондах».
- Цензура запрещённой лексики и неэтичных формулировок.
- Возвращение пользователю.

### 2.9. Журналирование

- Запрос пользователя (обезличенный).
- Контекст (ID источников).
- Ответ LLM (обезличенный).
- Время отклика.
- Версия промпта и моделей.
- Журнал хранится 12 месяцев, затем обезличивается.

## 3. Метрики качества

### 3.1. Точность (precision)

Доля ответов, содержащих корректное цитирование с правильной атрибуцией.

Целевое значение: ≥ 95% к M12.

### 3.2. Полнота (recall)

Доля запросов, на которые сервис нашёл релевантный ответ в корпусе.

Целевое значение: ≥ 80% к M12 при размере корпуса 10 000+ источников.

### 3.3. «Галлюцинации» (hallucination rate)

Доля ответов, содержащих несоответствие контексту (выдуманные факты, неверные ссылки).

Целевое значение: ≤ 1% к M12.

### 3.4. Время отклика (latency)

p50: ≤ 1.5 сек, p95: ≤ 3 сек.

### 3.5. Пользовательская оценка

NPS и thumbs-up/thumbs-down под каждым ответом.

## 4. Бенчмарк «золотой набор»

- 200 ручно подготовленных вопросов по русской классике, истории, культуре.
- Эталонные ответы согласованы с экспертом-филологом.
- Прогон еженедельно, отчёт сохраняется.
- При регрессии (падение precision или recall на > 5%) — релиз блокируется до разбора.

## 5. Защита от злоупотреблений

### 5.1. Anti-jailbreak

- Список патернов prompt-injection (например, «забудь предыдущие инструкции»).
- Фильтр на входе и на выходе LLM.
- При обнаружении — отказ с мотивированным сообщением.

### 5.2. Лимиты

- 60 запросов/минуту с IP.
- 1000 запросов/день для авторизованного пользователя.
- 50 000 запросов/день для одного B2B-кабинета.

### 5.3. Возрастные режимы

- 14+: дополнительные фильтры (шок-контент, насилие, эротика, психологические темы кризисного характера).
- 18+: стандартные фильтры.
- При обнаружении возраст-чувствительной темы в режиме 14+ — выдача с дисклеймером и ссылкой на профильную службу.

## 6. Версионирование

- Промпты — в репозитории `backend/prompts/`, версионированы.
- Модели — фиксируются в конфиге.
- Релиз RAG-пайплайна — отдельный тег.

## 7. Отказоустойчивость

- При недоступности YandexGPT — переход на GigaChat MAX.
- При недоступности обоих — сервис возвращает заранее подготовленные ответы по часто задаваемым темам и предупреждает пользователя.

## 8. MVP-имплементация

В MVP-прототипе:

- Корпус: 30+ источников из реестра (`03_корпус_реестр.md`).
- Поиск: BM25 (rank_bm25 на Python).
- LLM: моковая генерация на детерминированных шаблонах.
- Полный пайплайн (BM25 → шаблон → постпроцессинг) реализован в `backend/app/rag.py`.

## 9. План перехода в продакшен

1. M3: BGE-M3 эмбеддинги вместо BM25.
2. M4: Подключение YandexGPT 5 Pro.
3. M5: Reranking через cross-encoder.
4. M6: Подключение GigaChat MAX как резерв.
5. M9: Подготовка к нагрузке 10 000 запросов/час.
6. M12: Достижение целевых метрик качества.
