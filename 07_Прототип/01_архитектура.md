# Архитектура прототипа ЧитАИ

## Назначение

Документ описывает архитектуру MVP-прототипа ЧитАИ: компоненты, потоки данных, сервисы, безопасность.

## 1. Компоненты

### 1.1. Frontend
- React 18 + Vite + TypeScript.
- Tailwind CSS, shadcn/ui.
- 4 страницы: Куратор (чат), Тренажёр ЕГЭ, Кабинет учителя, Дашборд региона.
- API: REST к Backend.

### 1.2. Backend
- FastAPI (Python 3.11+).
- Эндпоинты: `/api/curator/chat`, `/api/curator/route`, `/api/quiz/items`, `/api/teacher/scenarios`, `/api/dashboard/metrics`, `/api/sources/search`, `/api/health`.
- Корпус источников — JSON-файл (на MVP), pgvector в продакшен-версии.
- RAG-пайплайн: vectorstore + retrieval + ranking + генерация (моковая на MVP).

### 1.3. Telegram-бот
- aiogram 3.
- Команды: `/start`, `/route`, `/help`, `/feedback`.
- HTTP-вызов backend для генерации ответов.

### 1.4. HTML-мокап
- Один файл `mockup.html` с inline CSS.
- Имитирует диалог Куратора с готовыми сценариями (3 варианта).
- Не требует backend.

### 1.5. База данных (продакшен)
- PostgreSQL 15 + pgvector.
- Таблицы: `users`, `sessions`, `routes`, `route_items`, `messages`, `quiz_items`, `quiz_attempts`, `feedback`, `sources`, `embeddings`.

### 1.6. Хранилище объектов
- Yandex Object Storage (S3-совместимый).
- Файлы: иллюстрации, PDF, аудио.

### 1.7. Кэш
- Redis (продакшен) — очереди, сессии, кэширование выдачи.

## 2. Поток данных (RAG)

```
[Запрос пользователя]
        ↓
[Безопасность: фильтр по PII / запрещённым темам]
        ↓
[Эмбеддинг запроса (mock на MVP, BGE-M3 или Yandex Embeddings в продакшене)]
        ↓
[Векторный поиск top-K в pgvector / mock dictionary на MVP]
        ↓
[Reranking (CE / cross-encoder) — опционально]
        ↓
[Сборка контекста + промпт]
        ↓
[Вызов LLM (YandexGPT 5 Pro / GigaChat MAX)]
        ↓
[Постобработка: проверка ссылок, защита от утечки PII, фильтр галлюцинаций]
        ↓
[Ответ пользователю + источники]
```

На MVP-прототипе LLM-вызов заменён детерминированной моковой генерацией на основе шаблонов и ключевых слов.

## 4. Безопасность

- TLS 1.3 для всех соединений.
- Аутентификация: JWT (HMAC-SHA256), refresh token.
- Хранение ключей: Yandex KMS.
- 2FA для административных кабинетов.
- Журналирование действий: Yandex Audit Trails.
- Anti-bot: capcha при подозрительной активности.
- Rate-limiting: 60 запросов/минуту с IP.
- Anti-jailbreak guardrails на входе и выходе LLM.

## 5. Локализация данных

- Все персональные данные граждан РФ — в дата-центрах РФ.
- Резервные копии — в РФ.
- Трансграничная передача — отсутствует.

## 6. Отказоустойчивость

- Активный регион Yandex Cloud + тёплый резерв SberCloud.
- Пер-сервисный health-check (`/api/health`).
- Авто-failover для критичных сервисов.
- RPO ≤ 5 минут, RTO ≤ 1 час (целевые значения для продакшена).

## 7. CI/CD

- GitHub Actions / Yandex DevOps.
- Линтеры: ruff (Python), eslint (JS), prettier.
- Тесты: pytest, vitest.
- Безопасность: bandit, npm audit.
- Развёртывание: Docker + Kubernetes (Yandex Managed Kubernetes) / Fly.io для MVP.

## 8. Логирование и мониторинг

- Application logs: structured JSON (loguru).
- Метрики: Prometheus + Grafana.
- Трассировка: OpenTelemetry.
- Алерты: алертинг по error rate, latency, доступности.
- Аудит ПД: Yandex Audit Trails (обязательно).

## 9. Версионирование

- Семантическое: MAJOR.MINOR.PATCH.
- Релизы — по тегам Git.
- Backwards-совместимые API изменения — в MINOR.

## 10. Disaster recovery

- Резервные копии БД: ежедневные + WAL архивирование.
- Хранение: 30 дней.
- Восстановление до точки во времени (PITR).
- Тестовое восстановление — раз в квартал.

## 11. Доступность для пользователей с ограниченными возможностями

- WCAG 2.1 AA.
- Поддержка скринридеров.
- Контрастность.
- Альтернативный текст для изображений.
- Управление с клавиатуры.

## 12. MVP-ограничения

В MVP-прототипе (этот репозиторий):

- LLM заменён моковой генерацией по шаблонам.
- Эмбеддинги — упрощённые (TF-IDF / BM25 для MVP, в продакшене — BGE-M3 + Yandex Embeddings).
- Корпус — 30–50 источников из public domain (русская классика).
- База — JSON-файл вместо PostgreSQL.
- Бот — заглушка, отвечает по фиксированным сценариям.
- Дашборд — синтетические данные.
