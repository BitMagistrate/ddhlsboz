# Этический протокол ЧитАИ

## 1. Назначение

Этический протокол описывает принципы, по которым ЧитАИ генерирует ответы, обрабатывает запросы и взаимодействует с пользователями. Протокол связан с Кодексом этики в сфере искусственного интеллекта (Альянс в сфере ИИ, 2021), Указом Президента РФ №490 от 10.10.2019 и нацпроектом «Культура».

## 2. Принципы

### 2.1. Прозрачность

- Каждый ответ сопровождается ссылкой на источник.
- При отсутствии данных в фондах сервис отвечает: «Не нашёл данных в фондах».
- Версии моделей, промптов и пайплайнов фиксируются, доступны B2G-заказчикам по запросу.

### 2.2. Достоверность

- Бенчмарк «золотой набор» из 200 вопросов, еженедельный пересчёт.
- Целевая доля «галлюцинаций» — ≤ 1% к M12.
- Двойная проверка для чувствительных тем (литературоведение, история, спорные вопросы) с использованием двух LLM (YandexGPT + GigaChat) и сравнения ответов.

### 2.3. Защита персональных данных

См. `01_политика_ПД.md`, `06_реестр_ПД.md`.

- Минимизация ПД.
- Локализация в РФ.
- Шифрование, KMS, аудит.

### 2.4. Недискриминация

- Качество ответов одинаковое вне зависимости от региона, школы, дохода, гендера.
- Платный paywall не закрывает базовые функции.
- Возрастные режимы — прозрачные, единые правила для всех пользователей.
- Тренировочные данные — сбалансированные по полу, регионам, этническим группам.

### 2.5. Защита от злоупотреблений

- Anti-jailbreak guardrails (фильтрация запросов и ответов).
- Тематические фильтры на медицину, психологию, юр. советы — выдача с дисклеймером и рекомендацией к специалисту.
- Дополнительные фильтры для несовершеннолетних: исключение шок-контента, насилия, суицидальной тематики.
- Журнал инцидентов с публикуемой базой знаний (без персональных данных).

### 2.6. Открытость стека

- Документация моделей, промптов, схемы данных, RAG-пайплайна доступна B2G-заказчикам.
- Экспорт пользовательских данных в JSON.
- Поддержка внешнего аудита.
- Использование сертифицированных российских облачных провайдеров.
- Open-source компоненты в стеке (FastAPI, PostgreSQL, pgvector, aiogram).

## 3. Ситуации отказа

Сервис отказывается отвечать в следующих случаях:

1. Запрос на медицинский диагноз, рецепт, дозировку.
2. Запрос на юридическую консультацию по конкретному делу пользователя.
3. Запрос на психотерапевтическую помощь при кризисном состоянии — выдача контактов экстренных служб.
4. Запрос на инструкции по причинению вреда (себе или другим).
5. Запрос на политическую агитацию, оправдание экстремизма или терроризма.
6. Запрос на разжигание ненависти по национальному, религиозному, гендерному признаку.
7. Запрос на эротический или порнографический контент.
8. Запрос с использованием персональных данных третьих лиц без их согласия.

В случае отказа сервис:

- сообщает причину отказа в нейтральной форме;
- при необходимости — предоставляет ссылки на профильные службы (МЧС, телефон доверия, профильные специалисты).

## 4. Реакция на инциденты

### 4.1. Распознавание инцидента

- Жалоба пользователя через форму обратной связи.
- Жалоба внешнего эксперта или СМИ.
- Внутренняя проверка в рамках еженедельных аудитов.
- Срабатывание автоматических фильтров.

### 4.2. Цикл реагирования

| Этап | Срок | Ответственный |
|---|---|---|
| Распознавание и регистрация | 4 часа | Дежурный CTO/CEO |
| Внутреннее расследование | 24 часа | CTO + Куратор контента |
| Уведомление пользователя/заказчика | 48 часов | CEO |
| Уведомление Роскомнадзора (при утечке ПД) | 24 часа от обнаружения | CEO |
| Публичное сообщение в базе знаний | 5 рабочих дней | CEO |

### 4.3. Журнал инцидентов

Ведётся в защищённом репозитории. Структура: дата, тип, описание, причина, решение, статус.

### 4.4. Анализ причин

Каждый инцидент разбирается на retrospective-сессии команды. Изменения в промптах, фильтрах, правилах — фиксируются в системе версионирования.

## 5. Внешний этический совет (M6+)

5.1. К моменту масштабирования (после 6 месяцев) Оператор формирует внешний этический совет из 3–5 экспертов:

- филолог-литературовед;
- специалист по информационной безопасности;
- юрист по защите ПД и авторскому праву;
- педагог-методист;
- представитель пользовательского сообщества (опционально).

5.2. Совет собирается ежеквартально. Результат — рекомендации по корректировке протокола.

## 6. Обучение сотрудников

6.1. Новые сотрудники проходят инструктаж по этическому протоколу в течение первой недели.

6.2. Раз в год — повторное обучение и аттестация.

6.3. Все сотрудники подписывают NDA и обязательство соблюдения этического протокола.

## 7. Прозрачность для пользователя

7.1. Сервис явно сообщает пользователю:

- что ответы генерируются ИИ-моделью на основе фондов РГБ/НЭБ и открытых источников;
- что ответы могут содержать неточности и требуют проверки;
- что для медицинских, юридических, психологических вопросов следует обратиться к специалисту;
- что пользователь вправе оставить обратную связь (like/dislike, комментарий).

## 8. Метрики этики

| Метрика | Целевое значение M12 |
|---|---|
| Доля «галлюцинаций» в ответах | ≤ 1% |
| Доля отказов в ответах с указанием причины | 100% от обоснованных отказов |
| Среднее время реакции на жалобу | ≤ 24 часа |
| Количество разобранных инцидентов | 100% зарегистрированных |
| NPS B2C | ≥ 45 |
| NPS B2G | ≥ 55 |
